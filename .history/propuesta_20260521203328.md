# Interfaces de Compiladores Aumentadas con IA para Accesibilidad en Entornos de Programación

## Idea Central de la Propuesta

La propuesta busca transformar el rol tradicional del compilador en entornos de programación accesibles.

En lugar de considerar al compilador únicamente como un traductor de código fuente hacia código ejecutable, se plantea utilizar sus representaciones semánticas internas como base para construir sistemas de accesibilidad más robustos e inteligentes.

La idea principal es:

> El compilador ya comprende formalmente el código, pero actualmente no comunica esa comprensión de forma accesible para todos los usuarios.

La investigación propone integrar:

* representaciones internas del compilador,
* mecanismos de accesibilidad estructural,
* y modelos de inteligencia artificial (LLMs),

para generar explicaciones accesibles, navegación estructural y diagnósticos comprensibles orientados especialmente a desarrolladores con discapacidad visual.

---

# Motivación

Las herramientas actuales de accesibilidad en programación presentan varias limitaciones:

* dependen principalmente de interfaces gráficas,
* utilizan soluciones externas (plugins o capas de accesibilidad),
* no aprovechan directamente la semántica interna del compilador,
* y continúan comunicando información compleja de forma visual o lineal.

Trabajos como CodeTalk identifican cuatro barreras fundamentales:

* discoverability,
* glanceability,
* navigability,
* alertability.

Estas barreras surgen porque:

* los IDEs modernos están diseñados principalmente para interacción visual,
* y el código suele consumirse línea por línea en lugar de estructuralmente.

---

# Gap de Investigación

La literatura actual aborda principalmente:

* accesibilidad desde la interfaz de usuario,
* navegación multimodal,
* herramientas de voz,
* o asistentes basados en IA generativa.

Sin embargo, existe poca investigación orientada a integrar simultáneamente:

1. representaciones semánticas del compilador,
2. accesibilidad estructural,
3. y modelos de IA para explicación adaptativa.

La propuesta busca cubrir esta brecha.

---

# Cambio de Paradigma Propuesto

## Modelo tradicional

```text
Código → Léxico → Sintáctico → AST → Semántico → Diagnósticos → IDE visual
```

La información semántica generada (AST, tabla de símbolos, errores) se descarta o reduce a mensajes lineales que el usuario debe interpretar sin soporte estructural.

---

## Modelo propuesto

```text
Código Fuente
    │
    ▼
Analizador Léxico  ──→  flujo de tokens
    │
    ▼
Analizador Sintáctico  ──→  AST
    │
    ▼
Analizador Semántico  ──→  Tabla de Símbolos + AST decorado
    │
    ▼
Generación de CFG  ──→  Grafo de Flujo de Control
    │
    ▼
Capa de Interpretación IA (LLM + semántica formal)
    │
    ▼
Interfaz Accesible Adaptativa (explicaciones, navegación, diagnósticos)
```

La accesibilidad deja de depender exclusivamente de la interfaz gráfica y pasa a fundamentarse en el conocimiento formal del compilador. Cada etapa expone sus estructuras internas hacia la capa de IA, que las usa como contexto verificado y libre de ambigüedad.

> “Compilador Aumentado con IA”

* interpretar semánticamente el código,
* generar explicaciones accesibles,
* permitir navegación estructural,
* y asistir procesos de comprensión y depuración.

---

# Componentes Fundamentales

## 1. Capa Semántica del Compilador

Se utilizarán estructuras internas como:

* AST (Abstract Syntax Tree),
* CFG (Control Flow Graph),
* tablas de símbolos,
* análisis semántico,
* diagnósticos del compilador.

Estas estructuras representan la “fuente de verdad” del sistema.

---

## 2. Capa de Interpretación Basada en IA

Los LLMs no trabajarán únicamente sobre texto plano.

En cambio:

* recibirán información semántica proveniente del compilador,
* reduciendo ambigüedad y alucinaciones.

Funciones esperadas:

* explicación de funciones,
* resúmenes semánticos,
* interpretación de errores,
* adaptación del nivel de detalle,
* explicación estructural del flujo del programa.

Ejemplo:

Entrada del compilador:

* variable fuera de scope.

Salida esperada:

> “La variable x no está disponible en este contexto porque fue declarada dentro de otro bloque.”

---

# Diferencia Frente a Sistemas Tradicionales

La propuesta NO busca simplemente:

* agregar voz,
* lectores de pantalla,
* o plugins visuales.

El aporte principal es:

> utilizar directamente la comprensión formal del compilador para producir accesibilidad.

La IA actúa como:

* mediador semántico,
* no como reemplazo del compilador.

---

# Relación con CodeTalk

CodeTalk constituye la principal referencia conceptual.

Aportes de CodeTalk:

* identifica barreras fundamentales,
* introduce navegación estructural,
* mejora accesibilidad en IDEs.

Limitación principal:

* opera principalmente a nivel de interfaz.

La propuesta actual extiende esa idea:

* trasladando la accesibilidad hacia el núcleo semántico del compilador,
* e incorporando IA explicativa.

# Metodología Propuesta

## Desarrollo del prototipo

### Lenguaje objeto: MiniLang

Para demostrar la arquitectura propuesta de forma controlada y reproducible, se diseña **MiniLang**: un lenguaje imperativo de tipado estático construido desde cero en Python 3.

MiniLang incluye:
* tipos primitivos: `int`, `float`, `bool`, `string`
* declaración de variables tipadas: `int x = 5;`
* definición de funciones: `func int suma(int a, int b) { return a + b; }`
* estructuras de control: `if/else`, `while`, `for`
* instrucciones de I/O integradas: `print(...)`, `read(...)`
* operadores aritméticos, relacionales y lógicos completos
* soporte a comentarios de línea (`//`) y bloque (`/* */`)

El compilador de MiniLang está estructurado en fases clásicas:

```text
Lexer → Parser → AST → [Semántico → CFG → Tabla de Símbolos]
         (fase 1)              (fase 2 — trabajos futuros)
```

La elección de un lenguaje propio (en lugar de Python AST, Tree-sitter o Clang) permite:
1. controlar completamente las estructuras internas expuestas a la capa IA,
2. producir diagnósticos pedagógicamente ricos,
3. demostrar que la arquitectura es agnóstica del lenguaje fuente.

### Tecnologías

* Lenguaje de implementación: Python 3.11+
* Sin dependencias externas para el núcleo del compilador
* Pruebas: `pytest`
